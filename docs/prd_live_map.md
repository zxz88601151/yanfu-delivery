## 1. 产品目标与核心指标

### 产品目标

通过骑手实时上报路况 + AI 交叉验证，构建配送区域路况热力图，帮助骑手避开拥堵/障碍区域，提升配送效率和体验，同时为动态定价模块提供红区因子输入。

### 核心指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 上报→确认响应时间 | ≤ 30 秒 | 从第 3 人上报到红区生成的时间 |
| 热力图更新响应 | ≤ 1 分钟 | 新红区确认后，热力图在 1 分钟内更新 |
| 验证准确性 | ≥ 95% | 已验证事件中真实事件的比例 |
| 骑手上报参与率 | ≥ 15% | 活跃骑手日均至少上报 1 次的比例 |
| 虚假上报率 | ≤ 3% | 经确认的虚假上报占总上报比例 |
| 红区影响区配送超时下降 | ≥ 10% | 红区标记后该区域配送超时率对比 |

## 2. 骑手上报规则（P0 细化）

### 2.1 上报入口与流程

**骑手 APP 交互路径**：

```
接单页面 → 底部悬浮按钮 "上报路况" → 弹出分类选择面板
  ├─ 🚧 修路施工
  ├─ 🚫 封路禁行
  ├─ 🛗 电梯故障（主要针对写字楼）
  ├─ 🚪 小区门禁难进
  ├─ ⚠️ 道路拥堵 >50 米
  └─ 📝 其他（手动输入描述）
```

- **必填项**：路况分类 + GPS 坐标（自动获取）
- **选填项**：文字描述（最多 200 字）、图片（最多 3 张，每张 ≤ 5MB，支持拍照/相册）
- **自动附加**：骑手 ID、上报时间、当前订单 ID（若有）

### 2.2 上报频率限制

| 限制项 | 规则 | 说明 |
|--------|------|------|
| 单骑手频率 | 同一分类每 **5 分钟** 最多 1 条 | 防止同一人重复刷屏 |
| 单日总量 | 每位骑手每日最多 **20 条** | 防止恶意批量上报 |
| 空间去重 | 同一位置 50 米范围内已有同类上报（未过期）则提示"该路况已有骑手上报" | 避免重复 |
| 新骑手限制 | 注册 < 3 天的骑手每日仅有 5 次上报额度 | 降低新骑手作弊风险 |

**超出限制时的响应**：
```json
{
  "code": 3001,
  "message": "上报过于频繁，请 5 分钟后再试",
  "data": {
    "next_available_at": "2025-06-19T12:10:00Z",
    "daily_remaining": 8,
    "reason": "same_category_rate_limited"
  }
}
```

### 2.3 上报权重计算逻辑

每条上报附带一个 **初始信任权重** `w`，用于 AI 交叉验证时加权判断：

```
w = rider_credit_weight × gps_accuracy_weight × evidence_weight
```

| 子权重 | 因子 | 取值范围 | 说明 |
|--------|------|----------|------|
| `rider_credit_weight` | 骑手信用分等级 | 0.6 ~ 1.2 | 青铜=0.8, 白银=0.9, 黄金=1.0, 钻石=1.2；新骑手(<50单)=0.6 |
| `gps_accuracy_weight` | GPS 精度 | 0.7 ~ 1.0 | 精度 ≤ 10m=1.0, ≤ 30m=0.9, ≤ 50m=0.8, >50m=0.7 |
| `evidence_weight` | 是否有图片证据 | 1.0 或 1.2 | 有图片=1.2，无图片=1.0 |

**示例**：钻石骑手（1.2）× GPS 精度 8m（1.0）× 有图片（1.2）= **权重 1.44**

> 权重用于交叉验证累加：当某位置的所有上报权重之和达到阈值（默认 **2.5**）时，标记为已验证。

## 3. AI 交叉验证策略（P0 细化）

### 3.1 验证流程

```
Step 1: 骑手 A 上报 → 存入 ai_road_reports (status=0)
Step 2: Verification Engine 每 15 秒扫描 status=0 的新记录
Step 3: 查询该位置 100 米半径内所有待验证上报
Step 4: 计算累计权重 w_total = Σ w_i
Step 5: 判断 w_total ≥ VERIFY_THRESHOLD (默认 2.5)?
         ├─ YES → 生成红区记录，更新 status=1，触发 road_report.verified 事件
         └─ NO  → 继续等待更多人上报（上报超过 30 分钟仍未达标则自动标记为"低置信度"并降级处理）
```

### 3.2 验证阈值策略

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 验证权重阈值 `VERIFY_THRESHOLD` | **2.5** | 累计权重达此值即确认（约等于 3 名普通骑手无图上报）|
| 空间匹配半径 | **100 米** | 以此半径搜索附近同类上报 |
| 超时未确认时长 | **30 分钟** | 超过此时间权重仍不足，标记为"低置信度" |
| 低置信度处理 | 加权值降至 0.3 | 低置信度上报仅作为辅助参考，不触发红区 |

**阈值设计说明**：
- 3 名普通骑手（权重各 0.8~1.0）无图上报 → 权重约 2.4~3.0 → 刚好跨过 2.5 阈值
- 2 名黄金骑手（各 1.0）且有图（各 1.2）→ 2.4 → 仍需第 3 人确认
- 1 名钻石骑手（1.2）+ 有图（1.2）+ 高精度（1.0）= 1.44 → 另需至少 1 名普通骑手（1.0）+ 才能验证

### 3.3 GPS 轨迹校验（防作弊）

**P1 补充**：在 `report-handler.js` 中增加 GPS 轨迹校验逻辑：

```
校验规则：
1. 检查骑手上报前 3 分钟内 GPS 轨迹是否经过上报点 50 米范围内
2. 若轨迹未经过 → 标记为"轨迹不匹配"，权重 × 0.3
3. 若轨迹经过 → 正常计算权重
4. 连续 3 次轨迹不匹配 → 该骑手当日报废所有上报，触发运营告警
```

> 轨迹数据从一期已有的骑手 GPS 位置表中获取。

### 3.4 上报合并规则

当同一位置（50 米内）已有同类型上报且状态为"待验证"时：

- 新上报不创建新记录，而是在已有记录上增加 `verified_count` 并更新权重累加
- 存在相同上报记录时返回 `{ message: "该路况已有骑手上报，感谢您的确认", verified_count: N }`
- 若已有记录已经验证通过（status=1），直接返回 `{ message: "该路况已被验证，获得快速确认积分" }`（给予较低激励）

## 4. 热力图生成规则（P0 细化）

### 4.1 颜色分级标准

`heatmap-generator.js` 根据 `ai_verified_conditions` 表中的 `difficulty_level` 和 `radius` 生成热力图数据：

| 颜色 | Level | 配送难度 | 判定条件 | 建议行为 |
|------|-------|----------|----------|----------|
| 🟢 绿色 | 0 | 畅通 | 无活跃红区 | 正常配送 |
| 🟡 黄色 | 1 | 轻微 | 修路施工、门禁难进，影响半径 ≤ 100m | 留意绕行建议 |
| 🟠 橙色 | 2 | 中度 | 道路拥堵 ≥50m、电梯故障，影响半径 100~200m | 建议绕行，延长预估时间 5min |
| 🔴 红色 | 3 | 严重 | 封路禁行、极端路况，影响半径 ≥ 200m | 强制避让，延长预估时间 10~15min |

### 4.2 热力图数据聚合规则

```javascript
// 热力图返回的数据结构（按 GeoHash 精度 6 级聚合）
{
  "tiles": [
    {
      "geo_hash": "wx4g0f",           // GeoHash 精度 6 级（约 1.2km × 0.6km）
      "center": { "lng": 116.397, "lat": 39.908 },
      "level": 2,                      // 0=绿 1=黄 2=橙 3=红
      "color": "#FF6B35",              // 前端直接使用的颜色值
      "conditions": [                   // 该区域内的活跃路况
        {
          "type": 1,                   // 1=修路
          "severity": "moderate",
          "radius": 150,
          "reported_count": 5
        }
      ],
      "updated_at": "2025-06-19T12:05:00Z"
    }
  ],
  "cached_at": "2025-06-19T12:05:00Z",
  "expires_in_seconds": 60
}
```

### 4.3 更新频率

| 触发方式 | 说明 |
|----------|------|
| 事件触发 | 新红区确认（`road_report.verified`）→ 立即更新受影响 GeoHash 区域 |
| 定时刷新 | node-cron 每 **2 分钟** 全量重算一次（兜底，清理过期数据）|
| API 查询 | `GET /heatmap` 请求时按需返回最新缓存数据（缓存 TTL = 60 秒）|

## 5. 红区避让逻辑（P1 细化）

### 5.1 红区生命周期

| 阶段 | 触发条件 | 状态 | 行为 |
|------|----------|------|------|
| 初始 | 验证通过 | active | 正常生效，影响热力图 + 路径规划 |
| 降级 | 超过 12 小时未收到新上报 | degrading | difficulty_level 降 1 级，影响半径缩小 50% |
| 过期 | 超过 24 小时 | expired | 从活跃红区中移除，触发 `road_report.expired` 事件 |
| 续期 | 过期前 2 小时内有新的确认上报 | 重置为 active | 生命周期重新计时 24 小时 |

### 5.2 影响半径与难度级别

| report_type | 默认半径 | 默认 difficulty_level | 说明 |
|-------------|----------|----------------------|------|
| 1=修路施工 | 150m | 2 (橙色) | 通常为局部施工，绕行可行 |
| 2=封路禁行 | 300m | 3 (红色) | 完全无法通行，必须避让 |
| 3=电梯故障 | 100m | 2 (橙色) | 影响写字楼配送效率 |
| 4=门禁难进 | 80m | 1 (黄色) | 需等待进入，轻微延误 |
| 5=道路拥堵 | 200m | 2 (橙色) | 拥堵超过 50 米路段 |
| 6=其他 | 100m | 1 (黄色) | 运营手动定级 |

### 5.4 路径避让策略

`auto-avoid.js` 的核心逻辑：

```javascript
function isAffectedByRedZone(orderRoute, verifiedConditions) {
  // 1. 将订单路径拆分为线段序列
  // 2. 对每条线段，检查是否与任何活跃红区的圆形范围相交
  // 3. 若相交，计算绕行距离增量
  // 4. 返回 { affected: true, extra_distance: 500m, extra_time: 5min, zones: [...] }
}

function suggestAvoidRoute(origin, destination, redZones) {
  // 1. 使用 @turf/turf 计算原始最短路径
  // 2. 若路径经过红区，计算绕行路径（绕过红区缓冲区 50m）
  // 3. 返回绕行路径 + 预估时间对比
}
```

**避让优先级**：

| 场景 | 策略 |
|------|------|
| 红区（Level 3）阻路 | 强制避让，路径引擎自动规划绕行路线 |
| 红区（Level 2）阻路 | 建议避让，标注"建议绕行，省约 X 分钟" |
| 红区（Level 1）附近 | 提示"前方有施工"但不强制避让 |
| 用户指定的配送地址在红区内 | 仍可配送，标注"配送地址位于红区，配送可能延迟" |

## 6. API 设计细化

### 6.1 API 端点总览

| 方法 | 路径 | 权限 | 说明 | 优先级 |
|------|------|------|------|--------|
| **POST** | `/reports` | 骑手端 | 骑手上报路况 | **P0** |
| **GET** | `/reports` | 运营端 | 获取路况上报列表（含分页/筛选） | **P1** |
| **GET** | `/heatmap` | 骑手端/调度端 | 获取配送难度热力图 | **P0** |
| **GET** | `/conditions` | 调度端/运营端 | 获取已验证路况列表 | **P1** |
| **POST** | `/conditions/:id/expire` | 运营端(admin) | 手动过期一条红区 | **P2** |
| **GET** | `/routes/avoid-advice` | 骑手端 | 获取某路径的红区避让建议 | **P1** |
| **GET** | `/stats` | 运营端(admin) | 活地图统计仪表盘数据 | **P2** |

## 7. 激励积分规则（P1 细化）

### 7.1 积分规则总表

| 行为 | 积分 | 说明 |
|------|------|------|
| 首次上报（基础） | **+5 分** | 无论最终是否验证通过 |
| 帮助确认他人上报 | **+3 分** | 在已有待验证记录上二次确认时 |
| 验证通过奖励 | **+10 分** | 该上报最终触发红区生成时，所有参与上报的骑手均获得 |
| 快速验证奖金 | **+5 分** | 在首条上报后 5 分钟内达到阈值，所有参与者额外获得 |
| 首次上报触发红区 | **+15 分** | 首位上报者若其上报触发了红区生成 |
| 附带图片 | **+2 分** | 上报时附带图片的额外奖励 |
| 每日上限 | **+60 分/天** | 单个骑手每日通过上报最多获得 60 分 |
| 虚假上报 | **-20 分** | 运营确认虚假后扣除，并标记信用 |
| 连续造假 | **-50 分 + 禁报 7 天** | 3 次及以上虚假上报 |

### 7.2 积分用途

| 用途 | 所需积分 | 说明 |
|------|----------|------|
| 积分排行榜展示 | — | 骑手 APP"活地图贡献榜"周榜/月榜 |
| 兑换小礼品 | 200 分起 | 矿泉��、防晒霜、手机支架等骑手福利品（对接 CarbonCredit 兑换体系）|
| 信用分加成 | 100 分 = +1 信用分 | 每月结算一次，上限 +20 分（对接 CreditPassport）|
| 优先派单权重提升 | 高贡献骑手自动获得 | 在上报活跃度前 10% 的骑手，派单优先级提升 5% |

### 7.3 虚假上报判定与处罚流程

```
运营后台 → 路况详情页 → "标记为虚假"按钮
  ├─ 系统自动扣除该上报者 20 分
  ├─ 记录虚假上报次数（ai_road_reports 扩展字段）
  ├─ 达到 3 次 → 自动进入"限制上报"状态（禁报 7 天）
  └─ 通知骑手（APP 推送）"您的一条上报被判定为虚假，已扣除 20 分"
```

运营端判定虚假上报后，关联的已验证红区自动删除（`ai_verified_conditions` 中移除），触发 `road_report.expired` 事件。

## 9. 更新待确认问题（附推荐答案）

| # | 原始问题 | 推荐答案 | 理由 |
|---|----------|----------|------|
| 1 | **验证阈值**：需要多少人确认才判定为真实事件？ | **权重 ≥ 2.5**（约为 3 名普通骑手） | ① 3 人机制是常见众包验证的标准折中方案；② 引入加权机制（信用等级+图片+GPS），让高信骑手单人贡献更大权重；③ 阈值可在运营后台动态配置 |
| 2 | **红区生命周期**：默认有效期多长？ | **24 小时自动过期**，12 小时自动降级 | ① 多数临时性路况（施工/拥堵/门禁）24 小时内会变化；② 12 小时降级机制避免红区"突然消失"的突兀感；③ 临近过期前有续期窗口（2 小时内新确认可重置）|
| 3 | **激励上限**：骑手每日通过上报最高获得多少积分？ | **60 分/天** | ① 防止刷分，保证公平性；② 约相当于每天 4-6 条有效上报的奖励，符合预期参与度；③ 60 分约可兑换 0.6 元价值的礼品，平衡激励成本 |
| 4 | **GPS 轨迹校验是否强制执行？** | **是**，作为 P1 实现 | ① 轨迹校验收效比高（防作弊核心手段）；② 一期已有骑手 GPS 位置表，数据获取无额外成本；③ 轨迹不匹配仅降权不拒绝，避免误伤 |
| 5 | **红区避让的优先级是否高于距离最优？** | **Level 3 红区强制避让 > 距离最优**；Level 2 建议避让 | ① 封路禁行（Level 3）完全无法通行，避让是刚性需求；② 中度路况（Level 2）绕路成本通常低于硬闯延误；③ 允许骑手自主选择，提供备选路线 |
| 6 | **是否允许用户选择"接受红区"（不避让）？** | **允许** | ① 部分骑手熟悉本地路况，可能知道更小的小路绕行方案；② 接单时提供绕行建议+接受红区选项；③ 选择接受红区时标注预计增加的时间 |

## 10. 新增补充需求

### 10.1 P1 新增 — 路况上报批量经营工具

运营后台支持以下批量操作：

| 操作 | 触发条件 | 影响 |
|------|----------|------|
| 批量确认 | 选中多个待验证上报 | 跳过 AI 验证，直接生成红区（用于已知的官方路况信息）|
| 批量忽略 | 选中疑似虚假上报 | 标记为已忽略，不上报者扣分 |
| 批量过期 | 选中多个活跃红区 | 立即失效，触发 `road_report.expired` |
| 手动创建红区 | 运营后台手动输入 | 输入坐标/半径/类型/等级，直接写入 `ai_verified_conditions`（用于大型活动/官方封路通知）|

### 10.2 P1 新增 — 官方路况公告接入

对接市政/交管部门 API（或运营手动录入），直接生成高置信度红区：

- 来源标记为"官方"（`report_type` 扩展值为 0）
- 权重直接设为最大值 3.0（单人即可触发验证）
- 红区标记特殊图标（🏛️ 官方标记）
- 过期时间根据公告日期设定

### 10.3 P2 新增 — 骑手贡献排行榜

| 周期 | 展示范围 | 奖励 |
|------|----------|------|
| 日榜 | 全平台前 10 名 | 展示性的"今日路况先锋"徽章 |
| 周榜 | 全平台前 20 名 | 200 积分 + "活地图达人"徽章 7 天 |
| 月榜 | 全平台前 50 名 | 1000 积分 + "城市鹰眼"徽章 + 优先派单权重提升 5% |

### 10.4 P2 新增 — 红区影响分析报表

运营后台"红区分析"页面：

| 维度 | 展示内容 |
|------|----------|
| 红区分布热力 | 全城红区聚集区域（哪里红区最多）|
| 红区类型占比 | 修路/封路/电梯等各类占比趋势 |
| 红区 vs 配送超时 | 红区标记前后该区域配送超时率对比 |
| 骑手上报活跃度 | 各区域上报骑手分布、上报响应时间 |
| 激励积分消耗 | 每日激励积分发放总量、人均积分 |

### 10.5 新增 P2 — WebSocket 实时推送

| 事件名 | 触发时机 | 推送对象 | 说明 |
|--------|----------|----------|------|
