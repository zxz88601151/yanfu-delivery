## 1. 产品目标重申

基于运力供需比、天气、时段、距离、订单密度 5 大因子，实现配送费的透明动态浮动。用价格杠杆调节供需关系——运力紧张时价格上涨抑制需求、激励骑手接单；运力充裕时价格回落，鼓励用户下单。**核心指标**：峰值订单平滑度 + 用户价格满意度 + 配送费收入平衡。

## 2. 5 因子定价模型 — 详细业务规则

### 2.1 综合计算公式

```
最终配送费 = 标准配送费 × 综合浮动系数
综合浮动系数 = base(1.0) × supply_demand_factor × weather_factor × time_factor × distance_factor × density_factor
浮动金额 = 最终配送费 - 标准配送费
```

> 所有因子以 `1.0` 为基准（无影响），>1.0 上调，<1.0 下调。
> 综合浮动系数经上下限裁剪后生效。

### 2.2 各因子定义

#### 因子 1：运力供需比（权重最高）

| 条件 | 因子值 | 业务含义 |
|------|--------|----------|
| 供需比 ≥ 2.0（运力充裕） | 0.85 ~ 0.95 | 轻度降价，鼓励下单 |
| 1.0 ≤ 供需比 < 2.0（正常） | 0.95 ~ 1.05 | 微调或无调整 |
| 0.5 ≤ 供需比 < 1.0（紧张） | 1.05 ~ 1.30 | 涨价抑制需求 |
| 供需比 < 0.5（严重紧张） | 1.30 ~ 1.50 | 大幅涨价，紧急调节 |

**供需比计算公式**：
```
供需比 = 当前区域活跃骑手数 / 当前区域待配送订单数
```
- 活跃骑手数：当前处于"空闲"或"配送中"状态的骑手总量
- 待配送订单数：已创建但尚未分配骑手的订单 + 正在配送中的订单
- 数据来源：EventBus 从 PrePosition 模块获取的 `surge.prediction.ready` 事件 + 实时订单表查询

**映射函数**（建议实现方式）：
```javascript
function calcSupplyDemandFactor(ratio) {
  if (ratio >= 2.0) return 0.90;                    // 运力充裕
  if (ratio >= 1.0) return 1.0 - (ratio - 1.0) * 0.1; // 线性过渡 1.0→0.95
  if (ratio >= 0.5) return 1.0 + (1.0 - ratio) * 0.6; // 线性过渡 1.0→1.30
  return 1.50;                                       // 严重紧张封顶
}
```

#### 因子 2：天气系数

| 天气等级 | 条件 | 因子值 | 说明 |
|----------|------|--------|------|
| 优良 | 晴/多云 | 1.0 | 无影响 |
| 轻度 | 小雨/小雪/微风 | 1.05 ~ 1.10 | 轻度上涨 |
| 中度 | 中雨/中雪/大风 | 1.15 ~ 1.25 | 中度上涨 |
| 重度 | 大雨/大雪/暴雨/暴雪 | 1.30 ~ 1.50 | 大幅上涨，结合上下限 |

- 数据来源：weather-client.js 调用第三方天气 API（由 config 配置 API Key）
- 根据配送目的地坐标的当前天气实况判定
- 缓存策略：天气数据缓存 30 分钟（天气变化较慢）

#### 因子 3：时段系数

| 时段 | 时间范围 | 因子值 | 业务逻辑 |
|------|----------|--------|----------|
| 凌晨低谷 | 00:00-06:00 | 1.20 ~ 1.50 | 运力稀缺，高溢价 |
| 早餐 | 06:00-09:00 | 1.0 | 正常水平 |
| 午间高峰 | 11:00-14:00 | 1.10 ~ 1.20 | 用餐高峰，需求旺盛 |
| 下午茶 | 14:00-17:00 | 0.90 ~ 0.95 | 低谷，降价促销 |
| 晚间高峰 | 17:00-21:00 | 1.05 ~ 1.15 | 下班高峰 |
| 夜宵 | 21:00-24:00 | 1.10 ~ 1.20 | 夜间运力紧张 |

- 实际取值可在运营后台的 `ai_price_configs` 表中动态配置（config_key = `TIME_FACTOR_MAP`）

#### 因子 4：距离系数

| 配送距离 | 因子值 | 说明 |
|----------|--------|------|
| ≤ 1km | 0.90 | 短距离补贴，鼓励接单 |
| 1km ~ 3km | 0.95 ~ 1.0 | 正常范围 |
| 3km ~ 5km | 1.0 ~ 1.10 | 中等距离上涨 |
| 5km ~ 10km | 1.10 ~ 1.25 | 长距离溢价 |
| ≥ 10km | 1.25 ~ 1.50 | 超长距离大幅溢价 |

- 使用 @turf/turf 计算起点到终点的实际路线距离（非直线距离）
- 长距离订单配送费上涨 = 激励骑手接远单

#### 因子 5：订单密度系数

| 条件 | 因子值 | 说明 |
|------|--------|------|
| 低密度（< 5 单/平方公里） | 0.90 ~ 0.95 | 订单稀疏，降价以吸引更多订单聚拢 |
| 正常（5~20 单/平方公里） | 1.0 | 无调整 |
| 高密度（20~50 单/平方公里） | 1.05 ~ 1.15 | 集中区域顺手单多，小幅溢价 |
| 极高密度（> 50 单/平方公里） | 1.15 ~ 1.25 | 爆单区域，价格杠杆拉满 |

- 订单密度 = 过去 15 分钟内该区域新增订单总数 / 区域面积（平方公里）
- 区域划分使用 Geohash 精度 6 级（约 1.2km × 0.6km）

### 2.3 浮动上下限约束（P0）

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 单次最大上浮金额 | +5 元 | 确保涨价不过于极端 |
| 单次最大下浮金额 | -3 元 | 降价幅度限制，保护平台收入 |
| 综合浮动系数上限 | 1.50 | 对应最高涨价 50% |
| 综合浮动系数下限 | 0.85 | 对应最高降价 15% |

- 以上参数均可在 `ai_price_configs` 表中动态配置
- **极端天气保护**：当天气因子 ≥ 1.30（大雨/暴雪等级）时，单独启用**天气封顶保护**，单次浮动上限改为 +8 元（而非默认 +5 元），确保骑手有充分激励同时用户感受可控

### 2.4 更新频率

**推荐方案**：5 分钟滚动窗口聚合计算，配合事件触发实时更新。
- **定时更新**：每 5 分钟由 node-cron 触发一次全区域定价系数重算
- **事件触发**：当 PrePosition 模块发出 `surge.prediction.ready` 事件时，即时触发受影响区域的定价系数更新
- **实时查询**：用户端 `GET /estimate` 接口每次请求时，基于最新缓存的定价系数实时计算最终报价（系数缓存不过期则不重复计算各因子）

## 3. API 设计预期

### 3.1 API 端点总览

所有接口均挂载于 **`/api/v2/ai/dynamic_pricing`**，遵循统一响应格式 `{ code: 0, data: {...}, message: "success" }`。

| 方法 | 路径 | 权限 | 说明 | P0/P1 |
|------|------|------|------|-------|
| **GET** | `/estimate` | 用户/匿名 | 估算配送费（下单前核心接口） | **P0** |
| **GET** | `/zone` | 调度端 | 获取区域定价系数列表 | **P0** |
| **GET** | `/config` | 运营端(admin) | 获取定价配置参数 | **P1** |
| **PUT** | `/config` | 运营端(admin) | 更新定价配置参数 | **P1** |
| **GET** | `/logs` | 运营端(admin) | 获取定价日志（含各因子明细） | **P2** |

### 3.2 核心接口详情

## 5. 更新待确认问题（附推荐答案）

| # | 原始问题 | 推荐答案 | 理由 |
|---|----------|----------|------|
| 1 | **动态定价更新频率**：实时计算还是按 5/15 分钟窗口聚合？ | **5 分钟窗口 + 事件触发** | ① 实时计算对数据库和天气 API 压力过大；② 5 分钟窗口与 PrePosition 预测模块对齐（surge.prediction.ready 每 5 分钟触发）；③ 用户体验上 5 分钟内价格变化是可接受的 |
| 2 | **极端天气封顶保护**：是否启用？封顶值如何？ | **启用**。当天气因子 ≥ 1.30 时，单次上浮上限从 +5 元放宽至 **+8 元** | ① 极端天气骑手出勤意愿低，需要更强激励；② 用户对极端天气涨价接受度较高；③ +8 元封顶对应约 60% 涨幅（以 5 元基准计），既可激励骑手又不至于引发客诉 |
| 3 | **各因子的权重是否需要统一调整入口？** | 所有因子采用**乘法模型**（非加权求和），通过 `ai_price_configs` 表的因子区间配置间接控制权重 | 乘法模型更符合定价直觉：任一因子极端时都会显著影响最终价格。运营可通过调整各因子的值域范围来间接控制其影响力 |
| 4 | **定价日志保留期限？** | 保留 **90 天**，超期归档至冷存储 | 满足运营分析需求，同时控制 `ai_price_logs` 表数据量 |

## 6. 新增补充需求

### 6.1 P1 细化 — 定价区域热力图数据聚合

**细化说明**：`GET /zone` 接口返回的数据需按时段缓存，调度后台每 5 分钟轮询刷新。响应数据需包含：
- `composite_factor` 综合系数（供热力图着色用：红 >1.2、黄 0.95~1.2、绿 <0.95）
- `supply_demand_ratio` 原始供需比（供调度员参考）
- 每个因子各自的贡献值（方便排查"哪个因子在驱动涨价"）

### 6.2 P1 新增 — 价格影响分析报表

| 需求项 | 说明 |
|--------|------|
| 报表维度 | 按区域/时段/天气等级交叉分析 |
| 指标 | 平均配送费、平均浮动金额、浮动订单占比、用户取消率 |
| 展示方式 | 柱状图 + 趋势折线图，支持导出 CSV |
| 数据来源 | `ai_price_logs` 表聚合查询 |
| 触发方式 | 运营后台手动刷新 + 每日自动生成快照 |

### 6.3 P2 细化 — 用户价格提醒

**细化说明**：
- 用户可在下单页点击"降价提醒我"按钮，设置目标配送费金额
- 系统通过 WebSocket 推送（若用户在线）或 APP 推送（若离线）提醒
- 提醒触发条件：当用户所在区域定价系数回落至指定阈值以下
- 每个用户最多同时设置 3 个提醒（防滥用）
- 提醒有效期：24 小时自动过期

### 6.4 P2 细化 — A/B 测试框架

**细化说明**：
- 支持对不同区域/时段/用户群组设置不同的定价参数配置
- 通过 `ai_price_configs` 表的 config_key 命名约定区分 A/B 组：`TEST_A_surge_cap_up`, `TEST_B_surge_cap_up`
- 运营后台创建 A/B 测试时需指定：实验名称、涉及区域、参数差异、起止时间、目标指标（如订单量变化率）
- 测试结束后自动生成对比报告

### 6.5 新增 P1 — 异常降级策略（熔断机制）

当某一因子获取失败时（如天气 API 超时），系统不应中断整个定价流程，而是：

| 场景 | 处理策略 |
|------|----------|
| 天气 API 超时 | 使用该区域最近一次缓存的天气数据；若无缓存，weather_factor 回退为 1.0 |
| 供需比计算失败 | 使用 PrePosition 模块的上一次预测数据；若无数据，supply_demand_factor 回退为 1.0 |
| 数据库连接异常 | 从 node-cache 读取最近一次定价配置缓存；若无缓存，使用代码内置默认值 |
| 全部因子获取失败 | 回退为标准配送费 × 1.0（即原价），记录异常日志并触发告警 |

### 6.6 新增 P2 — WebSocket 实时推送

当定价系数发生显著变化（综合浮动系数变动 ≥ 0.10）时，通过 WebSocket 推送消息：

```json
{
  "event": "dynamic_pricing.zone_updated",
  "data": {
    "district_id": 12,
    "district_name": "朝阳区CBD",
    "old_factor": 1.20,
    "new_factor": 1.50,
    "changed_at": "2025-06-19T12:00:00Z",
    "reason": "surge_prediction_triggered"
  }
}
```

## 9. 与上下游模块的交互约定

| 上游模块 | 输入 | 方式 |
|----------|------|------|

|----------|------|------|
| 订单系统 | `estimate` API → 最终配送费 | HTTP 响应 |
| 骑手 APP | 接单卡片中的激励金额 | API 调用 |
| WebSocket 推送 | 定价系数变更通知 | WS Push |
