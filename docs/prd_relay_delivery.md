> **文档版本**：v1.0

## 1. 产品目标与核心指标

### 产品目标

对超远距离配送订单自动拆分为多段接力，由多位骑手分段完成，缩短单个骑手单次配送距离，提升远距离配送时效，减少骑手因长途单拒单率。

### 核心指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 远距离订单（>5km）配送时长 | 降低 **50%** | 相比单骑手配送的预计时长 |
| 骑手接力单接单率 | ≥ **80%** | 推送给骑手的接力单被接受的比例 |
| 接力点交接耗时 | ≤ **3 分钟** | 从 A 骑手到达接力点到 B 骑手交接完成的平均时间 |
| 接力配送成功率 | ≥ **95%** | 发起接力后最终成功送达的比例 |
| 用户端满意度 | ≥ **4.0/5.0** | 接力配送订单的用户评分 |

## 2. 拆单逻辑（P0 细化）

### 2.1 触发条件

订单提交后，由 `split-engine.js` 评估是否适合拆分为接力配送。**必须同时满足以下所有条件**：

| 条件 | 阈值 | 说明 |
|------|------|------|
| 配送距离 | **> 5km** | 原始 PRD 已有约定 |
| 预估配送时长 | **> 40 分钟** | 单骑手时长超出合理范围 |
| 订单金额 | ≥ **20 元** | 低金额单拆分后每段配送费过低，骑手无意愿 |
| 接力点可用 | 配送路径上存在至少 1 个可用接力点 | 详见第 3 节接力点选择 |
| 运力充足 | 接力点前后区域均有空闲骑手 | 依赖 T05 PrePosition 运力数据 |

> **不触发拆分的场景**：订单中有生鲜/易腐品类标记时，优先单骑手配送（避免交接延误导致品质问题）；用户明确勾选了"不接受接力配送"时。

### 2.2 分段规则

```
总距离 D km，分段段数 N
规则：
  1. 每段距离范围：2km ≤ 单段距离 ≤ 4km
  2. 最大分段数：3 段（即最多 3 名骑手接力）
  3. 分段数计算公式：
     N = min(3, max(2, floor(D / 3.5)))
  4. 各段距离尽量均匀，最后一段略短（减少尾段配送压力）
```

**示例**：
| 总距离 | 段数 | 分段方式 | 说明 |
|--------|------|----------|------|
| 5~7 km | 2 段 | 3km + 2~4km | 最常见的 2 段接力 |
| 7~10 km | 3 段 | 3km + 3km + 1~4km | 超过 7km 启用 3 段 |
| 10 km 以上 | 3 段 | ~3.3km × 3 | 上限 3 段，不再增加 |

### 2.3 拆单触发时机

```
Step 1: 用户提交订单 → 订单系统调用 POST /relay_delivery/split
Step 2: SplitEngine 评估条件（距离/金额/接力点/运力）
Step 3: 
  ├─ 条件满足 → 生成接力方案 → 分配骑手（异步等待骑手确认）
  └─ 条件不满足 → 返回 { splittable: false, reason: "..." } → 退化为普通单
Step 4: 骑手确认完成后 → 更新 ai_relay_orders 状态 → 触发 relay.order.created 事件
```

> **关键**：拆单评估是**同步**的（下单后立即判断），但骑手分配是**异步**的（等待骑手接单确认）。若 5 分钟内未分配完所有段骑手，则该接力单降级为普通单（尝试分配一名骑手配送全程）。

## 3. 最优接力点选择（P0 细化）

### 3.1 接力点库

接力点来源于以下渠道（按优先级排序）：

| 优先级 | 来源 | 示例 | 说明 |
|--------|------|------|------|
| 🥇 首选 | 平台自有驿站 | 盐阜配送驿站 | 已在合作，空间稳定，有充电设施 |
| 🥈 次选 | 合作便利店/商户 | 美宜佳、罗森等 | 需签署合作协议，提供包裹暂存 |
| 🥉 备选 | 公共设施 | 地铁站出口、公交枢纽 | 无需合作但环境不可控，仅限非恶劣天气 |
| ⚠️ 最低 | 虚拟接力点（路侧交接）| 路边约定坐标 | 仅用于紧急情况（如 B 骑手就近等待）|

### 3.2 接力点选择算法

`split-engine.js` 选择接力点的逻辑：

```javascript
function selectRelayPoints(orderRoute, segments) {
  // 1. 计算各分段边界点（理想中的接力位置）
  // 2. 在边界点 500m 半径内搜索接力点库
  // 3. 按优先级排序（驿站 > 便利店 > 公共设施 > 虚拟点）
  // 4. 检查接力点当前是否可用（未被占用、营业中）
  // 5. 返回最优接力点列表
}
```

**选择标准**（按权重评分）：

| 标准 | 权重 | 说明 |
|------|------|------|
| 距理想边界点距离 | 30% | 越近越好，≤200m 满分 |
| 接力点类型优先级 | 25% | 驿站 10 分，合作商户 8 分，公共设施 5 分，虚拟 2 分 |
| 历史交接成功率 | 20% | 该点历史的交接完成率 |
| 周边骑手密度 | 15% | 该点 1km 内空闲骑手数 |
| 天气影响 | 10% | 户外接力点雨天扣分 |

### 3.3 接力点的地理数据

接力点在数据库中以单独表或配置形式维护（可在 `ai_price_configs` 或单独表中存储）：

```json
{
  "id": "RP001",
  "name": "盐阜配送·望京驿站",
  "type": "station",            // station | store | public | virtual
  "location": { "lng": 116.48, "lat": 39.99 },
  "address": "北京市朝阳区望京 SOHO T1 一层",
  "business_hours": { "open": "07:00", "close": "23:00" },
  "amenities": ["parking", "shelter", "charging"],
  "status": "active"
}
```

## 4. 骑手匹配策略（P0 细化）

### 4.1 分段骑手匹配规则

每段骑手的分配由 `handoff-manager.js` 调用 T05 PrePosition 的运力数据和 T06 CreditPassport 的信用数据综合决定：

| 匹配条件 | 前段骑手（段 1） | 中段骑手（段 2） | 后段骑手（段 3） |
|----------|-----------------|-----------------|-----------------|
| 当前位置 | 距商家 ≤ 1km | 距接力点 1 ≤ 1km | 距接力点 2 ≤ 1km |
| 信用等级 | ≥ 白银 (Level 2) | ≥ 白银 (Level 2) | ≥ 黄金 (Level 3) |
| 车辆类型 | 不限 | 不限 | 建议电动车（便于灵活走街串巷）|
| 派单权重 | 距离近 > 信用高 > 空闲时长 | 距离近 > 历史接单率 > 信用高 | 信用高 > 历史送达准时率 > 距离近 |

> **前段骑手**负责从商家到接力点 1，**后段骑手**从接力点 1 到顾客；若有 3 段则**中段骑手**负责接力点 1 到接力点 2。

### 4.2 推单与接单流程

```
Step 1: 系统计算最优骑手列表（每段 Top 3）
Step 2: 按优先级依次推送接单通知
Step 3: 骑手有 30 秒决定接受/拒绝
Step 4: 接受 → 锁定骑手；拒绝/超时 → 推送下一位
Step 5: 所有段骑手确认后 → 更新状态为"已分配"
```

**推单并行策略**：前段和后段骑手可以**同时推送**，无需等前段确认后再推后段。但当某段骑手未被分配时，系统有 **5 分钟** 的缓冲时间寻找替代骑手。

### 4.3 接力单的骑手端标识

骑手接单卡片上显眼标注：

```
┌───────────────────────────────────┐
│  🤝 接力配送 · 第 1 段/共 2 段     │ ← 黄色标签
│  ──────────────────               │
│  📍 商家 → 盐阜望京驿站            │
│  距离：3.2 km  预计：15 min        │
│  接力点到达时间：12:30              │
│  配送费：¥4.50                    │
│  ──────────────────               │
│  [  接单  ]   [  查看接力点  ]      │
└───────────────────────────────────┘
```

后段骑手可见前段骑手的实时位置和预计到达时间。

## 5. 交接流程（P0 细化）

### 5.1 正常交接流程

```
前段骑手                    后段骑手                      系统
   │                          │                          │
   │  ── PUT /arrive ────────→│                          │
   │  到达接力点，标记到达      │                          │
   │                          │                          │
   │  放置包裹/扫码确认         │                          │
   │                          │  ← 推送"请前往接力点取餐"  │
   │                          │                          │
   │                          │  ── PUT /arrive ────────→│
   │                          │  后段骑手到达接力点        │
   │                          │                          │
   │  ── PUT /handoff ──────→│  ← 确认取餐                │
   │  前段确认交接             │                          │
   │                          │                          │
   │                 系统更新状态为"交接完成"                 │
   │                          │                          │
   │                          │  ── 开始配送后半段 ──────→│
```

### 5.2 交接超时处理

| 场景 | 等待时间 | 处理方式 |
|------|----------|----------|
| 前段已到，后段未到 | 5 分钟 | 系统推送提醒后段骑手；同时搜索 500m 内备选骑手 |
| 后段未到超过 10 分钟 | 10 分钟 | 自动触发备选骑手调度；包裹暂存接力点（驿站场景）|
| 前段已到且无驿站存包 | 10 分钟 | 降级：前段骑手继续配送后半段（按全程计费）|
| 后段已到，前段迟到 | — | 后段等在接力点，前段到达后快速交接（后段等待时间计入配送时长补偿）|

### 5.3 交接确认方式

| 环境 | 确认方式 |
|------|----------|
| 驿站/合作商户 | 前段骑手扫码确认放入 → 后段骑手扫码确认取出（系统自动记录）|
| 公共/虚拟接力点 | 前后段骑手**面对面扫码确认**（前段出示二维码，后段扫描）+ GPS 位置校验收在 50m 内 |
| 异常情况（无法扫码）| 前段拍照上传包裹照片 → 运营后台确认 → 手动标记交接完成 |

## 6. 配送费拆账规则（P1 细化）

### 6.1 按距离比例拆账

```
总配送费 = F（来自订单系统，含动态定价浮动）
各段配送费 = F × (段距离 / 总距离) × 段难度系数
```

**段难度系数**：

| 段位置 | 系数 | 说明 |
|--------|------|------|
| 前段 | 1.0 | 基准，包含取餐等待时间 |
| 中段（若有）| 1.1 | 纯运输，无等待，小幅溢价 |
| 后段 | 1.2 | 包含末端配送（上楼/等待用户），最高系数 |

**计算示例**（总配送费 10 元，总距离 8km，分 2 段）：

```
前段：3.5km，系数 1.0
  金额 = 10 × (3.5/8) × 1.0 = 4.38 元
后段：4.5km，系数 1.2
  金额 = 10 × (4.5/8) × 1.2 = 6.75 元
  合计 ≈ 11.13 元 → 平台补贴 1.13 元（超出部分记作接力激励成本）
```

> **平台补贴规则**：当拆账后总额超过原配送费时，超出部分由平台承担（作为接力配送激励成本）。当拆账后总额低于原配送费时，按比例分配不截留差价。

### 6.2 异常情况拆账

| 场景 | 拆账规则 |
|------|----------|
| 前段完成，后段骑手迟到 → 前段继续配送 | 前段获得原配送费的 **80%**，后段无费用 |
| 前段完成，后段因故取消 → 备选骑手接替 | 前段正常计费，备选骑手获得后段费用的 **90%** |
| 前段迟到导致后段等待超时 | 前段扣减 **10%** 配送费补给后段 |
| 接力配送因用户取消 | 各段按已完成距离比例结算 |
| 商品在接力点丢失/损坏 | 责任判定前暂停费用结算，根据责任方扣费 |

## 7. 异常处理（P2 细化）

### 7.1 异常场景汇总

| 异常类型 | 触发条件 | 处理动作 | 优先级 |
|----------|----------|----------|--------|
| 前段骑手配送中取消 | 前段骑手接单后配送途中取消 | 搜索接力点 1km 内备选骑手；无合适则降级为单骑手订单 | P0 |
| 后段骑手接单后取消 | 后段骑手确认后配送前取消 | 立即推送备选骑手，前段暂缓出发或暂存接力点 | P0 |
| 后段骑手超时未到 | 前段到达后等待 > 10 分钟 | 自动调度备选骑手；备选不可用则前段继续配送全程 | P1 |
| 接力点不可用 | 接力点临时关闭/打烊 | 选择最近备用接力点；无备用则降级为单骑手 | P1 |
| 商品损坏/丢失 | 交接时发现异常 | 拍照留证 → 暂停接力 → 运营介入判定责任 | P2 |
| 天气突变 | 途中等恶劣天气 | 接力点转为驿站室内交接；户外接力点自动调整 | P2 |

### 7.2 异常接力自动恢复机制

```
前段取消 → anomaly-handler 检测触发
  ├─ 检查已完成配送距离占比
  ├─ <30% 总距离 → 整体取消该接力方案，重新寻找全程骑手
  ├─ ≥30% 总距离 → 保留已完成段，仅寻找替代该段的骑手
  └─ 若接力点驿站可存包 → 前段存包后离开，等待新骑手取包
```

### 7.3 顾客投诉溯源

用户投诉接力配送订单时，运营后台可查看：

| 分段 | 数据项 |
|------|--------|
| 前段 | 骑手信息、取餐时间、配送轨迹、接力点到达时间 |
| 接力点 | 交接时间、交接方式、包裹状态记录 |
| 后段 | 骑手信息、接力点出发时间、配送轨迹、送达时间 |

> 投诉责任判定依据：哪个分段超时/异常 → 对应骑手责任；接力点交接期间问题 → 平台/驿站责任。

### 8.3 预计到达时间（ETA）计算

```
总 ETA = 前段配送时长 + 交接缓冲时间（3 分钟）+ 后段配送时长
       + 末端上楼时间（2 分钟/km 步行）

各段 ETA 每分钟刷新一次，考虑：
  - 实时路况（对接 T04 活地图红区数据）
  - 骑手 GPS 行驶速度
  - 交接点排队情况（驿站高峰期 +2 分钟）
```

## 9. API 设计细化

### 9.1 端点总览

| 方法 | 路径 | 权限 | 说明 | 优先级 |
|------|------|------|------|--------|
| **POST** | `/split` | 系统内部 | 评估并创建接力配送方案 | **P0** |
| **GET** | `/orders/:id` | 用户/骑手端 | 获取接力配送详情 | **P0** |
| **PUT** | `/handoffs/:id/arrive` | 骑手端 | 骑手到达接力点标记 | **P0** |
| **PUT** | `/handoffs/:id/handoff` | 骑手端 | 接力交接确认 | **P0** |
| **GET** | `/progress/:order_id` | 用户端 | 用户端进度查询 | **P1** |
| **GET** | `/stations` | 骑手端 | 获取可用接力点列表 | **P2** |
| **POST** | `/stations` | 运营端(admin) | 新增接力点 | **P2** |

### 9.2 核心接口详情

#### POST /api/v2/ai/relay_delivery/split

> 拆单评估与接力方案创建

**Request**：

## 10. 更新待确认问题（附推荐答案）

| # | 原始问题 | 推荐答案 | 理由 |
|---|----------|----------|------|
| 1 | **接力配送的最低距离阈值**？ | **5km** 且 **预计时长 > 40 分钟** | ① 5km 是骑手接单意愿的明显分水岭；② 需要同时考虑时长（5km 拥堵 vs 5km 高速差异大）；③ 与架构文档已约定一致 |
| 2 | **最高分段数**？ | **3 段（3 名骑手）** | ① 每段 2~4km 覆盖最合理；② 超过 3 段协调成本激增（4 名骑手至少需要 3 个接力点）；③ 10km 以上 3 段各约 3.3km，体验可接受 |
| 3 | **接力点是否需要第三方合作协议**？ | **需要但非必须** | ① 驿站和合作商户优先，可保障包裹安全和交接体验；② 公共设施/虚拟点作为兜底方案，不依赖合作；③ 上线初期建议先自建 3~5 个驿站试点，再逐步拓展合作商户 |
| 4 | **接力配送费超出原配送费时，差价谁承担？** | **平台承担**（计入接力激励成本） | ① 接力配送本身降���了总配送时长，提升了用户体验；② 超出部分通常不大（约 10~15%）；③ 可设定月预算上限，超额时评估是否调整系数 |
| 5 | **是否有不让用户感知接力配送的需求？** | **需要向用户透传** | ① 透明度提升信任度；② 用户看到"接力配送"可理解等待时间变化；③ 但需用"🤝 极速接力"等正面话术包装，避免用户担心包裹安全 |
| 6 | **接力点等待超过多久触发备选方案？** | **前段等待 10 分钟** | ① 5 分钟发提醒，10 分钟触发备选；② 10 分钟对用户来说可接受（总时长仍优于单骑手）；③ 备选方案包括：存包驿站→找新骑手、前段继续配送 |
