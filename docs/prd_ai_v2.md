## 1. AI 盲盒配送 🎲

### 产品目标
通过 AI 将用户盲盒订单匹配至商家临期/低销量餐品（打折出售），实现用户省钱、商家清库存、平台促新三方共赢。

### 用户故事

| 角色 | 故事 |
|------|------|
| 普通用户 | 作为普通用户，我想要下盲盒单（仅填预算和口味偏好），以便用更低价格吃到惊喜餐品 |
| 商家 | 作为商家，我想要将临期/低销量餐品加入盲盒池，以便减少浪费并获取新客 |
| 新开店铺 | 作为新店店主，我想要通过盲盒配送获得首批曝光和订单，以便快速积累口碑和销量 |

### 需求池

| 优先级 | 需求 | 说明 |
|--------|------|------|
| **P0** | 盲盒订单创建流程 | 用户选择"盲盒配送"模式，填写预算范围、口味偏好（辣/清淡/甜等）、配送区域，AI 匹配后展示结果 |
| **P0** | AI 匹配引擎 | 根据预算、口味偏好、区域，从商家库存中筛选临期/低销量餐品，按折扣率排序推荐 |
| **P0** | 盲盒订单结算 | 按折扣价结算，平台补贴与商家分摊机制 |
| **P1** | 商家盲盒管理后台 | 商家可设置参与盲盒的餐品、折扣幅度、库存上限 |
| **P1** | 新店/新品首发推荐 | 新店/新品可标记为"首发"，优先进入盲盒匹配池 |
| **P2** | 盲盒评价与反馈 | 用户可评价盲盒餐品，帮助 AI 优化匹配精度 |
| **P2** | 盲盒社交分享 | 用户分享盲盒开箱结果到社交媒体，获平台积分奖励 |

### UI 设计稿说明

- **首页入口**：在"立即下单"按钮旁增加"盲盒配送 🎲"入口按钮，突出惊喜感
- **盲盒下单页**：表单含预算范围（滑块 10-50 元）、口味多选标签（辣/清淡/甜/酸/咸）、配送地址；底部"开启盲盒"大按钮带动效
- **盲盒结果页**：展示匹配餐品（含原价 vs 盲盒价对比），显眼"确认下单"按钮；倒计时限时确认
- **商家后台**：盲盒管理 Tab，含餐品列表、参与状态开关、折扣率设置、库存计数器

### 待确认问题

1. 盲盒餐品的折扣下限是否有约束？是否需设定平台最低折扣率（如不低于 5 折）？
2. 用户对盲盒结果不满意是否可退款？退款规则如何定义？

## 2. AI 预置运力 🔮

### 产品目标
利用 AI 预测模型提前预判爆单区域，在峰值到来前将骑手前置调度至该区域"等单"，缩短爆单时段配送时长。

### 用户故事

| 角色 | 故事 |
|------|------|
| 调度员 | 作为调度员，我想要系统自动显示"未来 30 分钟即将爆单区域"，以便提前安排骑手就位 |
| 骑手 | 作为骑手，我想要接到"前往 X 区域预置"的调度指令，以便在爆单前到达热点区域减少空跑 |
| 运营经理 | 作为运营经理，我想要查看预置运力的效果分析报表（爆单响应时长对比），以便评估 ROI |

### 需求池

| 优先级 | 需求 | 说明 |
|--------|------|------|
| **P0** | 爆单预测模型 | 基于历史订单数据 + 天气 + 时段 + 商圈活动，预测未来 30/60 分钟各区域订单密度热力图 |
| **P0** | 预置调度指令 | 系统自动向目标区域骑手推送"预置调度"任务，含建议位置、预计等待时长 |
| **P0** | 预置区域地图展示 | 调度后台热力图展示预测爆单区域，颜色渐变标识紧急程度 |
| **P1** | 骑手预置状态管理 | 骑手 APP 显示"预置中"状态，含预计爆单时间倒计时 |
| **P1** | 预置效果仪表盘 | 对比预置区域 vs 非预置区域的爆单配送时长、骑手空跑率 |
| **P2** | 自动补调机制 | 当预置骑手不足时，系统自动触发补充调度 |

### UI 设计稿说明

- **调度后台地图**：主界面为热力图，红色区域 = 预测爆单区；侧边栏列出预置区域及建议骑手人数
- **骑手 APP 预置卡片**：弹出调度卡"预置调度：X 商圈预计 15 分钟后爆单，请前往"，含导航按钮和接受/拒绝
- **效果报表页**：折线图对比预置前/后爆单高峰配送时长，表格展示各区域数据

### 待确认问题

1. 骑手拒绝预置调度是否影响信用分？预置调度的激励（额外补贴）如何设定？
2. 预测模型的更新频率是实时还是定时（如每 10 分钟）？

## 3. AI 协同配送 🤝

### 产品目标
将远距离大单拆分为多个接力段，由多名骑手分段配送，接力点交接，目标远距离配送时效提升 50% 以上。

### 用户故事

| 角色 | 故事 |
|------|------|
| 远距离用户 | 作为远距离下单的用户，我想要订单被分段配送，以便缩短等待时间 |
| 接力骑手 A | 作为骑手 A，我想要接单后只配送我负责的前半段到接力点，以便不离开我的常驻区域 |
| 接力骑手 B | 作为骑手 B，我想要在接力点无缝接单后配送后半段，以便高效完成远距离配送 |

### 需求池

| 优先级 | 需求 | 说明 |
|--------|------|------|
| **P0** | 接力配送拆单引擎 | 当配送距离超过阈值（如 5km），AI 自动评估是否拆分，计算最佳接力点和分段方案 |
| **P0** | 接力点管理 | 系统维护接力点库（商圈驿站、便利店、合作网点等），作为骑手交接物理位置 |
| **P0** | 骑手分段接单与交接 | A 骑手配送至接力点 → 确认到达 → B 骑手接单 → 继续配送；全程追踪 |
| **P1** | 接力配送费拆账 | 按分段距离比例自动分配配送费给各骑手 |
| **P1** | 用户端接力进度展示 | 用户可实时看到"骑手 A 正在配送前半段 → 接力点交接中 → 骑手 B 正在配送后半段" |
| **P2** | 异常接力处理 | 接力点 B 骑手未及时到达时，自动触发备选骑手调度或转为单骑手配送 |

### UI 设计稿说明

- **用户订单状态页**：分段进度条，标注"骑手 A → 接力点 → 骑手 B"，每个阶段显示骑手头像和位置
- **骑手接单页**：若为接力单，显眼标注"接力配送（前半段/后半段）"，含接力点地图标记和预计到达时间
- **调度后台**：接力单列表，显示各段骑手状态、接力点位置、预计总时长

### 待确认问题

1. 接力配送的最低订单金额或距离阈值应设为多少？
2. 接力点是否需要与第三方（便利店/驿站）签署合作协议？

## 4. AI 活地图 🗺️

### 产品目标
基于骑手上报 + AI 交叉验证，生成实时配送难度热力图（红区/绿区），调度时自动避开高难度区域，提升配送预测准确率。

### 用户故事

| 角色 | 故事 |
|------|------|
| 骑手 | 作为骑手，我想要上报遇到的配送障碍（修路、电梯故障等），以便其他骑手不再踩坑 |
| 调度系统 | 作为调度系统，我想要自动获取"活地图"数据，以便为订单规划避开红区的最优路线 |
| 运营 | 作为运营人员，我想要查看各区域配送难度趋势，以便联合物业/社区改善配送环境 |

### 需求池

| 优先级 | 需求 | 说明 |
|--------|------|------|
| **P0** | 骑手上报入口与流程 | 骑手 APP 内快速上报按钮（修路/封路/电梯故障/小区难进等分类），附位置和图片 |
| **P0** | AI 交叉验证引擎 | 同一位置收到 ≥3 名骑手上报后自动确认并生成红区标记 |
| **P0** | 配送难度热力图 | 调度后台展示实时热力图（红/黄/绿），红区自动影响路径规划权重 |
| **P1** | 红区自动避让调度 | 路径引擎自动避开红区，或标注额外配送时长 |
| **P1** | 上报激励机制 | 骑手上报获积分奖励，高贡献骑手排名展示 |
| **P2** | 红区生命周期管理 | 红区自动过期机制（如上报后 24 小时自动降级），支持运营手动清除 |

### UI 设计稿说明

- **骑手 APP 上报页**：底部悬浮按钮"上报路况"，点击弹出分类选项（修路/封路/电梯/门禁/其他），可拍照上传
- **调度后台热力图**：地图叠加层，红色渐变表示配送难度，点击红区显示上报详情和骑手数量
- **运营管理页**：红区列表 + 批量操作（确认/忽略/设置过期时间）

### 待确认问题

1. 骑手上报的真实性如何防作弊？是否需引入 GPS 轨迹校验？
2. 红区自动避让的优先级是否高于距离最优？是否允许用户选择"接受红区"？

## 5. AI 信用护照 🛂

### 产品目标
建立跨平台可携带的骑手信用体系，信用分基于准时率、客诉率、好评率、接单履约率，高分享有优先派单权，激励优质骑手、淘汰劣质骑手。

### 用户故事

| 角色 | 故事 |
|------|------|
| 优质骑手 | 作为高信用骑手，我想要优先获得优质订单（高单价、短距离），以便通过高信用获得实际收益 |
| 新骑手 | 作为新注册骑手，我想要了解信用分如何提升，以便尽快达到高信用等级 |
| 平台运营 | 作为平台运营，我想要根据信用分设置不同的骑手权限和派单权重，以便正向激励整个骑手生态 |

### 需求池

| 优先级 | 需求 | 说明 |
|--------|------|------|
| **P0** | 信用分模型 | 算法计算准时率（30%）、客诉率（25%）、好评率（25%）、接单履约率（20%），总分 0-1000 |
| **P0** | 信用分展示与等级 | 骑手 APP 信用分主页，等级划分（青铜/白银/黄金/钻石），升降级规则 |
| **P0** | 优先派单权 | 高信用骑手在同等条件下优先派单，优质订单（高配送费/短距离）优先分配 |
| **P1** | 信用扣分/加分规则 | 明确行为对应的信用分变动（拒单 -5、超时 -10、投诉成立 -20 等） |
| **P1** | 申诉与复核机制 | 骑手对扣分有异议可通过申诉流程提交证据，人工复核 |
| **P2** | 跨平台信用互通接口 | 定义标准化信用分 API，支持未来接入其他配送平台 |
| **P2** | 行业信用联盟 | 与行业协会合作制定统一信用标准 |

### UI 设计稿说明

- **骑手 APP 信用主页**：顶部信用分值（大字）和等级徽章，环状进度条展示四项维度得分，下方为近期信用变动时间线
- **派单效果页**：高信骑手可见"今日派单质量"统计（均价单、平均距离、接单率）
- **运营后台**：骑手信用列表、信用分布饼图、申诉管理队列

### 待确认问题

1. 新骑手初始信用分设为多少？（建议 600 分，经历 50 单后进入正式评估）
2. 跨平台互通的数据隐私和合规问题如何解决？

## 6. AI 动态定价 💰

### 产品目标
基于运力供需比、天气、时段、距离、订单密度等多因子，实现透明的配送费动态浮动，用价格杠杆调节供需，削峰填谷。

### 用户故事

| 角色 | 故事 |
|------|------|
| 用户 | 作为用户，我想要在下单前清楚看到配送费上涨原因和幅度，以便决定是否等待低价时段下单 |
| 骑手 | 作为骑手，我想要在运力紧张时段获得更高配送费，以便获得合理的劳动回报 |
| 运营经理 | 作为运营经理，我想要监控动态定价对订单量和运力匹配的影响，以便调优定价参数 |

### 需求池

| 优先级 | 需求 | 说明 |
|--------|------|------|
| **P0** | 动态定价模型 | 综合运力供需比、天气系数、时段系数、距离、订单密度 5 大因子计算配送费浮动系数 |
| **P0** | 用户端价格透明展示 | 下单前显眼展示"标准配送费 X 元 + 运力调节 +Y 元 = 应付 Z 元"，附原因说明 |
| **P0** | 浮动上下限约束 | 设置单次浮动上限（如 ±5 元），确保定价不过于极端 |
| **P1** | 骑手端收入展示 | 骑手接单前看到"含动态激励"的配送费总额，激励单展示"高峰激励 +X 元" |
| **P1** | 定价区域热力图 | 调度后台展示各区域当前运力供需比及对应定价系数 |
| **P2** | 用户价格提醒 | 用户可设置"配送费降至 X 元时通知我"，在价格回落时推送 |
| **P2** | A/B 测试框架 | 支持对不同区域/时段进行差异化定价策略 A/B 测试 |

### UI 设计稿说明

- **用户下单页**：配送费明细区域，标准费 + 浮动费分行展示，浮动部分用橙色标注并附"⚡当前运力紧张"图标
- **骑手接单页**：接单卡上显眼展示"高峰激励 +X 元"，与基础配送费分开展示
- **运营仪表盘**：定价系数热力图 + 时间轴折线图（订单量/运力/定价系数联动）

### 待确认问题

1. 动态定价的更新频率：实时计算还是按 5/15 分钟窗口聚合计算？
2. 极端天气下是否启用"封顶"价格保护机制？封顶值如何确定？

## 7. AI 碳积分 🌱

### 产品目标
基于配送距离 × 交通工具碳排系数追踪每单碳排放，通过积分兑换激励用户和骑手绿色行为，实现 ESG 合规 + 用户粘性 + 绿色出行激励。

### 用户故事

| 角色 | 故事 |
|------|------|
| 环保用户 | 作为关注环保的用户，我想要查看每单的碳减排数据，以便了解我的绿色消费贡献 |
| 骑手 | 作为骑手，我想要看到电动车相比摩托车的减碳对比，以便认可绿色出行的价值 |
| ESG 负责人 | 作为企业 ESG 负责人，我想要获取平台碳排放报告，以便满足合规披露要求 |

### 需求池

| 优先级 | 需求 | 说明 |
|--------|------|------|
| **P0** | 碳排放计算引擎 | 每单跟踪配送距离 × 交通工具碳排系数（电动车 0g/km、摩托车 80g/km、汽车 150g/km） |
| **P0** | 用户碳积分账户 | 用户端展示累计碳积分、碳减排量（kg CO₂）、积分明细 |
| **P0** | 积分兑换商城 | 优惠券、免配送费券等兑换商品，设定兑换比例（如 100 碳积分 = 1 元） |
| **P1** | 绿色时段激励 | 运力充足时段标记为"绿色时段"，用户此时段下单获额外碳积分 |
| **P1** | 骑手绿色排行 | 骑手端电动车 vs 摩托车碳排放对比榜，绿色骑手徽章 |
| **P1** | 碳足迹报告 | 月度/季度平台碳足迹报告（总减排量、绿色配送占比），支持导出 |
| **P2** | 碳积分社交分享 | 用户可分享碳减排成就到社交媒体 |
| **P2** | 企业碳抵消合作 | 与碳抵消平台合作，支持碳积分兑换为实际碳抵消认证 |

### UI 设计稿说明

- **用户碳积分主页**：顶栏显示总碳减排量（kg CO₂）+ 积分值；环状图展示"我的绿色贡献"
- **订单碳足迹卡片**：每单完成后展示"本单碳排放 Xg CO₂，相比摩托车减少 Yg"
- **绿色时段标记**：下单页时段选择器上标注"🌱 绿色时段 + 额外 N 积分"标签
- **骑手绿色排行**：列表展示骑手绿色排名（按电动车配送占比）、绿色徽章

### 待确认问题

1. 碳积分是否具有实际货币价值？兑换比例是否需与财务部门确认合规性？
2. 电动车 vs 摩托车的碳排系数是否需要第三方机构认证以保证 ESG 报告公信力？

## 附录：功能依赖关系与优先级排序

```
功能依赖关系图（建议实施顺序）：

第一优先级 ─┬─ AI 盲盒配送（独立，快速上线）
            ├─ AI 预置运力（依赖预测模型，一期已具备）
            ├─ AI 活地图（骑手上报功能独立）
            └─ AI 信用护照（独立，数据逐步积累）

第二优先级 ─┬─ AI 动态定价（依赖运力供需预测）
            ├─ AI 协同配送（依赖活地图 + 接力点网络）
            └─ AI 碳积分（独立，数据被动采集）
```
